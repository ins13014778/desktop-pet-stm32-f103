# 桌面宠物项目二开详细说明

这份文档不是泛泛地讲“项目结构”，而是从二次开发视角解释：

- 哪些文件最值得改
- 每个文件的职责是什么
- 文件内部哪些代码块最关键
- 新增功能时应该同时联动哪些文件

---

## 1. 先记住这条主线

这个项目的核心工作流非常简单：

1. `BlueTooth.c` 通过串口中断接收语音或蓝牙命令
2. 中断里修改全局状态，例如 `Action_Mode`、`Face_Mode`
3. `main.c` 主循环根据 `Action_Mode` 调用动作函数
4. `PetAction.c` 负责真正控制舵机完成动作
5. `Face_Config.c` 和 `OLED_Data.c` 负责把 `Face_Mode` 映射成 OLED 表情
6. `TIM3_IRQHandler()` 负责 LED 常亮或呼吸灯

所以几乎所有二开都能拆成下面三问：

- 怎么触发
- 触发后改哪个状态
- 这个状态最终由哪个函数执行

---

## 2. 最重要的可改文件

### `User/main.c`

这是整个工程的“动作调度中心”。

#### 主要职责

- 初始化舵机、OLED、蓝牙/语音串口
- 进入 `while(1)` 主循环
- 根据 `Action_Mode` 分发动作函数
- 在 `TIM3_IRQHandler()` 中执行 LED 灯效

#### 最关键的代码块

1. `main()` 初始化阶段

- `Servo_Init()`：初始化 PWM 和舵机控制底层
- `OLED_Init()`：初始化 OLED
- `BlueTooth_Init()`：初始化 USART1/USART3 和中断

2. `while(1)` 里的 `if / else if` 动作分发链

这段代码本质上就是一个简化状态机。

例如：

- `Action_Mode == 4` 时执行 `Action_advance()`
- `Action_Mode == 13` 时执行 `Action_Hello()`

如果你新增了一个动作，主循环必须知道“哪个状态值要调用哪个函数”，所以这里一定要改。

3. `TIM3_IRQHandler()`

这是 LED 的核心逻辑：

- `AllLed == 1 && BreatheLed == 0`：常亮
- `AllLed == 1 && BreatheLed == 1`：呼吸灯
- `AllLed == 0`：全灭

如果你要做新的灯效，比如闪烁、流水、双闪，这里是主入口。

#### 二开时什么时候改它

- 新增 `Action_Mode`
- 调整系统默认启动姿态
- 修改默认表情
- 增加新的 LED 模式

---

### `HardWare/BlueTooth.c`

这是项目最关键的“输入解析层”。

#### 主要职责

- 初始化 `USART1` 和 `USART3`
- 配置 GPIO、波特率、中断优先级
- 在 `USART1_IRQHandler()` 中解析语音命令
- 在 `USART3_IRQHandler()` 中解析蓝牙命令
- 维护多个全局状态变量

#### 这个文件里最重要的全局变量

- `Action_Mode`：当前动作模式
- `Face_Mode`：当前表情模式
- `SpeedDelay`：移动动作延时，越小越快
- `SwingDelay`：摇摆/摇尾速度
- `WeiBa`：尾巴开关
- `AllLed`：总灯开关
- `BreatheLed`：呼吸灯开关
- `Sustainedmove`：是否连续运动

#### 最关键的代码块

1. `BlueTooth_Init()`

负责配置：

- `USART1`：语音模块
- `USART3`：蓝牙模块
- 串口对应 GPIO
- 串口接收中断

如果以后你要换串口、换引脚、换蓝牙模块，第一处要看这里。

2. `USART1_IRQHandler()`

语音命令入口。

它的逻辑模式很固定：

- 收到一个命令字节
- 判断是哪个命令
- 修改 `Face_Mode`
- 调 `Face_Config()`
- 修改 `Action_Mode` 或灯光状态

3. `USART3_IRQHandler()`

蓝牙命令入口。

和语音逻辑基本一致，最大差异是：

- 蓝牙会把 `Sustainedmove = 1`
- 语音会把 `Sustainedmove = 0`

这就是为什么蓝牙更像“持续控制”，而语音更像“单次触发”。

#### 这里最值得注意的复杂点

当前代码多次调用 `USART_ReceiveData()` 直接比较命令值，没有先读到局部变量。

从工程稳定性角度，建议后续重构成：

```c
uint16_t cmd = USART_ReceiveData(USARTx);
switch (cmd) {
    ...
}
```

这样更清晰，也更安全。

#### 二开时什么时候改它

- 新增语音命令
- 新增蓝牙命令
- 修改已有命令和功能的映射关系
- 增加新的调速命令
- 增加新的表情切换逻辑
- 增加新的灯效控制命令

---

### `HardWare/PetAction.c`

这是动作实现层，也是二开最常改的文件之一。

#### 主要职责

- 通过控制 4 个主舵机和 1 个尾巴舵机实现所有动作
- 定义前进、后退、转向、摇摆、摇尾、打招呼、伸懒腰等动作函数

#### 文件内部怎么理解

你可以把它看成“动作脚本集合”。

每个动作函数其实都是：

- 让某个舵机转到某个角度
- 延时一段时间
- 再让另一个舵机转到另一个角度
- 按序重复

#### 重要动作类型

1. 静态姿态类

- `Action_relaxed_getdowm()`
- `Action_sit()`
- `Action_upright()`
- `Action_getdowm()`

特点：执行一次到位，主要决定“宠物摆什么姿势”。

2. 持续动作类

- `Action_advance()`
- `Action_back()`
- `Action_Lrotation()`
- `Action_Rrotation()`
- `Action_Swing()`
- `Action_SwingTail()`

特点：内部一般有 `while(Action_Mode == X)`，会持续执行，直到有新命令把 `Action_Mode` 改掉。

3. 单次表演类

- `Action_JumpU()`
- `Action_JumpD()`
- `Action_Hello()`
- `Action_stretch()`
- `Action_Lstretch()`

特点：执行一个完整动作序列后，通常会自动切回其他状态。

#### 文件里几个复杂代码块的含义

1. `while(Action_Mode == X)`

表示这个动作会持续跑，不是“一次性动作”。

2. `PAnumbers` 和 `Sustainedmove`

这组变量组合起来决定：

- 是走固定几步就停
- 还是一直走，直到蓝牙松手或新命令打断

3. `TiaoTurn` / `TiaoTurn2`

用于前跳、后跳时左右脚交替，避免每次都从同一边起步。

4. `WeiBa == 1` 的判断

表示如果尾巴开关打开，某些姿态动作结束后会自动转入尾巴动作模式。

#### 二开时什么时候改它

- 新增动作
- 优化动作姿态
- 调整动作节奏
- 改善动作是否流畅
- 调整哪个动作结束后切换到什么状态

---

### `HardWare/PetAction.h`

#### 主要职责

- 对外声明 `PetAction.c` 里的动作函数

#### 二开时什么时候改它

- 每次新增动作函数，都要在这里补声明

这个文件本身不复杂，但它是“让主循环能看到新动作”的必要一环。

---

### `HardWare/Face_Config.c`

这是“表情映射层”。

#### 主要职责

- 根据 `Face_Mode` 决定 OLED 显示哪张图

#### 最关键的代码块

```c
if(Face_Mode == 0) { ... }
else if(Face_Mode == 1) { ... }
...
```

这段代码本质上就是：

- 表情编号
- 对应图像资源

#### 当前含义

- `0`：睡眠
- `1`：发呆/盯眼
- `2`：开心
- `3`：狂热
- `4`：非常开心
- `5`：眼睛
- `6`：打招呼

#### 二开时什么时候改它

- 新增 `Face_Mode`
- 修改某个状态对应的表情
- 同一个动作想换成不同表情

---

### `HardWare/OLED_Data.c`

这是图像资源层。

#### 主要职责

- 存储表情图像数组
- 存储字模、字符图案等 OLED 数据

#### 最关键的代码块

- `const uint8_t Face_sleep[]`
- `const uint8_t Face_happy[]`
- `const uint8_t Face_hello[]`
- 其他 `Face_*` 数组

如果你要把现在的宠物脸换成你自己的脸、眼睛、表情动画，改的就是这些数组。

#### 二开时什么时候改它

- 替换已有表情图
- 新增表情图
- 新增图标或自定义显示素材

#### 注意点

这个文件非常占 Flash。

如果你新增很多表情、动画帧、图标，程序空间会明显变大。

---

### `HardWare/Servo.c`

这是舵机抽象层。

#### 主要职责

- 把“角度”转换成 PWM 比较值
- 屏蔽不同舵机方向差异

#### 最关键的代码块

例如：

```c
PWM_SetCompare1(Angle / 180 * 2000 + 500);
PWM_SetCompare2((180 - Angle) / 180 * 2000 + 500);
```

含义是：

- 左侧舵机直接映射
- 右侧舵机做反向映射

这能保证你在动作层写“90 度”时，左右两侧在机械含义上尽量一致。

#### 二开时什么时候改它

- 某个舵机方向反了
- 舵机中位不准
- 舵机整体角度区间要重新校准

---

### `HardWare/PWM.c`

这是 PWM 和定时器配置层。

#### 主要职责

- 初始化 TIM2 / TIM3
- 初始化 GPIO 复用输出
- 配置各个 PWM 通道
- 提供 LED 和舵机的 compare 设置函数

#### 关键资源映射

- `TIM2 CH1-CH4`：4 个主舵机
- `TIM3 CH1`：尾巴舵机
- `TIM3 CH3 / CH4`：LED1 / LED2

#### 二开时什么时候改它

- 换定时器
- 换 PWM 引脚
- 改 LED 通道
- 改 PWM 频率

这是硬件绑定层，改动后通常要一起检查：

- `Servo.c`
- 实物接线
- 主循环和动作层是否仍对应正确

---

### `HardWare/OLED.c`

这是 OLED 通用驱动层。

#### 主要职责

- 模拟 I2C
- 控制 OLED 初始化
- 提供图像、字符、图形绘制 API

#### 当前项目实际用得最多的函数

- `OLED_Init()`
- `OLED_Clear()`
- `OLED_ShowImage()`
- `OLED_Update()`

#### 二开时什么时候改它

- OLED 接线变更
- 要把显示从“纯图片”升级为“图片 + 文本”
- 要扩展更多显示能力

一般情况下，它不是最先需要改的文件。

---

### `HardWare/Delay.c`

#### 主要职责

- 提供阻塞式延时函数

#### 它为什么重要

因为动作流畅度高度依赖它。

`PetAction.c` 里大量的 `Delay_ms()` 决定了动作节奏。

#### 二开时什么时候改它

通常不建议动。

除非你准备把整个动作系统重构成非阻塞式状态机。

---

## 4. 常见新增功能应该怎么改

### 新增动作

要改：

- `HardWare/PetAction.c`
- `HardWare/PetAction.h`
- `User/main.c`
- `HardWare/BlueTooth.c`

逻辑顺序：

1. 写动作函数
2. 暴露函数声明
3. 让主循环能分发到它
4. 给它一个命令入口

---

### 新增语音或蓝牙命令

优先改：

- `HardWare/BlueTooth.c`

如果这个命令只是复用现有动作：

- 只改 `BlueTooth.c` 往往就够了

如果这个命令触发全新动作：

- 还要联动 `PetAction.c / PetAction.h / main.c`

---

### 新增表情

要改：

- `HardWare/OLED_Data.c`
- `HardWare/OLED_Data.h`
- `HardWare/Face_Config.c`
- `HardWare/BlueTooth.c` 或其他触发表情状态的地方

---

### 修改舵机方向或角度基准

如果只是某个动作不自然：

- 改 `PetAction.c`

如果是某个舵机在所有动作里都方向不对：

- 优先改 `Servo.c`

如果是引脚或定时器都变了：

- 改 `PWM.c`

---

### 新增灯效

要改：

- `User/main.c` 里的 `TIM3_IRQHandler()`

如果还要通过命令控制它：

- 联动 `HardWare/BlueTooth.c`

---

## 5. 最值得解释的复杂代码块

### 复杂块 1：主循环里的动作分发

这块虽然看起来只是长长的 `if / else if`，但它的实际作用是：

- 把“状态值”翻译成“具体动作函数”

它就是整个项目的中央调度表。

---

### 复杂块 2：串口中断里同时改 `Face_Mode` 和 `Action_Mode`

这说明系统设计思路是：

- 命令来了以后，不是先做动作后做显示
- 而是中断里立即切表情，并设定后续动作状态

也就是说，显示和动作在逻辑上是并行切换的。

---

### 复杂块 3：持续动作里的 `while(Action_Mode == X)`

这表示：

- 当前动作会持续执行
- 什么时候停，不由动作函数自己决定
- 而是由外部中断发来的新命令来改变 `Action_Mode`

这也是为什么蓝牙连续控制能成立。

---

### 复杂块 4：`Sustainedmove`

这个变量是“单次触发”和“持续触发”的分界线。

- 语音：一般一次说一个命令，所以置 `0`
- 蓝牙：往往是按住或连续发命令，所以置 `1`

动作函数里据此判断是执行几次后自动停下，还是持续执行。

---

### 复杂块 5：`Servo.c` 里的角度反向换算

这个逻辑的价值很大。

它把机械结构上的“左右方向相反”隐藏在底层驱动里，而不是把这种复杂性暴露给每一个动作函数。

这样动作层可以更专注于“姿态设计”，而不是“这边要正转，那边要反转”。

---

## 6. 二开建议顺序

如果你准备系统性读懂并改这个项目，推荐顺序是：

1. `HardWare/BlueTooth.c`
2. `User/main.c`
3. `HardWare/PetAction.c`
4. `HardWare/Face_Config.c`
5. `HardWare/OLED_Data.c`
6. `HardWare/Servo.c`
7. `HardWare/PWM.c`

这个顺序的好处是：

- 先看“怎么触发”
- 再看“怎么分发”
- 再看“怎么执行”
- 最后看“怎么显示”和“怎么落到硬件”

---

## 7. 开发时最容易踩的坑

- 只在 `PetAction.c` 新增动作，但忘了在 `main.c` 加状态分发
- 只在 `BlueTooth.c` 加命令，但忘了在 `PetAction.h` 声明新函数
- 换了 OLED 图但忘了在 `Face_Config.c` 建立映射
- 动作不对却直接改 `Servo.c`，导致所有动作一起被影响
- 改了 `PWM.c` 的引脚或通道，但没同步确认接线和舵机映射

---

## 8. 一句话总结

这个项目做二开时，最关键的理解方式不是“每个文件分别是什么”，而是：

- 命令从哪里进来
- 状态在哪里被改
- 主循环怎么根据状态调度
- 动作具体在哪里执行
- 显示和灯效怎么跟着状态变化

只要把这条链路弄清楚，你后面不管是加动作、加表情、加命令、改灯效，都会非常快。
